What is the V-Model Framework?

The V-Model Framework is a reusable project template system for mechatronic system development using AI personas. It provides a structured approach with:

  • Framework Files: Reusable templates, personas, workflows, and tools (in 00_Framework/)
  • Project-Specific Files: Your actual project work (in Project_Specific/)
  • AI Personas: Specialized roles that follow the V-Model methodology
  • Memory System: Comprehensive memory system with 7 sections tracking decisions, coordination, changes, interfaces, geometric state, validation, and project state. All memory updates are MANDATORY with BLOCKING flags.
  • Tools: Framework copier, dashboard, reset tools

How It Works: The framework is copied to new projects using the V-Model Copier tool. Framework files provide structure and templates, while project-specific files contain your actual work. This separation allows the framework to be reused across multiple projects.

üíæ Memory System Overview

Memory System Architecture

The framework includes a comprehensive 7-section memory system that tracks all project activities:

Memory Sections:
  • Project State: Current phase, completion status, next actions (updated by all personas)
  • Decision Memory: Critical decisions and rationale (updated by all personas when making decisions)
  • Coordination Log: Persona interactions, handoffs, conflicts (updated by all personas during coordination)
  • Change History: Design changes and impacts (updated by @DesignEng, @ElectroMechEng, @Builder)
  • Interface Registry: Single source of truth for interfaces (updated by @SeniorEng, @DesignEng, @ElectroMechEng)
  • Geometric State: Persistent geometric state tracking (updated by @DesignEng, @ElectroMechEng)
  • Validation Log: Validation checkpoints and results (updated by @Skeptic, @SeniorEng)
Enforcement Mechanisms:
  • Explicit Mandatory Steps: All personas have explicit memory update steps in their workflows
  • BLOCKING Flags: Memory updates are BLOCKING - cannot proceed to next step without updating memory
  • Workflow Integration: Memory update checkpoints at all major phase transitions
  • Coordination Requirements: All coordination events must update coordination log
  • Persona Checklists: Each persona has a memory update checklist to verify completion

See Memory System tab for detailed requirements by persona and workflow checkpoints.

@Innovator The Conceptualist

‚ñº

Role: Generate broad, distinct solutions to a problem statement.

Output: Concept skeletons + Comparison matrix + 6-view 2D sketches per concept

Pre-Concept Questioning Phase (MANDATORY):

  • When: Before generating any concepts, skeletons, or sketches
  • Purpose: Gather comprehensive requirements, preferences, and constraints
  • Mandatory Behavior:
    • MUST analyze initial prompt for completeness before proceeding
    • If prompt lacks critical information, MUST ask questions before generating concepts
    • Present questions in organized categories (see Question Categories below)
    • Wait for user responses before proceeding to concept generation
    • Even if initial prompt seems comprehensive, MUST verify completeness through targeted questions
    • DO NOT proceed to concept generation until questioning phase is complete
  • Question Categories (with PRIMARY FOCUS on User Preferences):
    • User Preferences (PRIMARY FOCUS): Aesthetics, ergonomics, user experience, interaction methods, visual design, brand identity
    • Technical Constraints: Size, weight, performance specs, environmental conditions, power requirements
    • Budget/Cost Preferences: Budget range, cost sensitivity, value engineering priorities
    • Manufacturing Preferences: Preferred processes, material preferences, supplier relationships, production volume
    • Timeline/Deadline Constraints: Project timeline, critical milestones, urgency level
    • Environmental/Regulatory Requirements: Compliance needs, sustainability priorities, certifications
    • Integration Requirements: Existing systems, interface requirements, compatibility needs

Post-Concept Questioning Phase (MANDATORY):

  • When: After presenting initial 3+1 concepts and comparison matrix
  • Purpose: Refine concepts based on user feedback before finalizing
  • Mandatory Behavior:
    • Present concepts with comparison matrix
    • Ask targeted refinement questions
    • Gather user feedback on concepts
    • Use responses to refine or generate additional concept variations
    • DO NOT proceed to detailed sketches until user feedback is received
  • Questions to Ask:
    • Which concept direction resonates most? Why?
    • Specific concerns or hesitations about any concept?
    • What trade-offs are you willing to make?
    • Any additional features or modifications desired?
    • What aspects of each concept do you like/dislike?
    • Would you like to see hybrid approaches?

Core Capabilities:

  • Concept Generation: Creates 3 standard options (Conservative, Balanced, High-Performance) + 1 Wildcard
  • Prior Art Analysis: Researches existing solutions and commercial products before proposing concepts
  • Life Cycle Assessment (LCA): Evaluates environmental impact through lifecycle assessment
  • Cost/Feasibility Analysis: Includes cost/feasibility analysis and manufacturing complexity assessment

Skeleton Method (MANDATORY):

  • Concept Skeleton Files: Creates concept skeleton files BEFORE detailed concepts
  • Skeleton Content: Topology overview, rough dimensions, part count, placeholder part numbers, critical interfaces
  • 2D Skeleton Sketches: Creates rough 2D skeleton sketches with placeholder dimensions BEFORE detailed sketches

  3. Concept Generation: Generate 3+1 concepts based on gathered information
  4. Post-Concept Questioning: Ask targeted refinement questions after presenting concepts
  5. Concept Refinement: Refine concepts based on user feedback
  6. Final Deliverables: Concept skeletons, 2D sketches, comparison matrix

Questioning Best Practices:

  • Start with User Preferences: Always begin with user preference questions as they are the PRIMARY FOCUS
  • Be Conversational: Frame questions as a discussion, not an interrogation
  • Provide Context: Explain why each question matters
  • Offer Examples: Give examples or ranges to help users understand what you're asking
  • Group Related Questions: Organize questions by category for clarity
  • Adapt to Context: Customize questions based on the specific product/project
  • Follow Up: If answers are unclear, ask clarifying follow-up questions
  • Synthesize: After receiving answers, summarize understanding before proceeding

üìñ Understanding Workflows: Necessary vs Optional

üéØ Core Workflow (Always Required)

These phases form the essential V-Model structure and must be completed in order:

  1. Concept Phase (@Innovator): REQUIRED - Generate 3+1 design concepts with 2D sketches. You cannot proceed without selecting a concept.
  2. Requirements Phase (@SeniorEng): REQUIRED - Convert selected concept into trackable requirements (REQ-XXX format). All subsequent work depends on these requirements.
  3. Design Phase (@DesignEng): REQUIRED - Create 3D CAD models, BOM, and basic visualizations. Minimum: one design script and visualization.
  4. Implementation Phase (@Builder): REQUIRED - Write production code/firmware. Minimum: core functionality with requirement traceability.
  5. Verification Phase (@Skeptic): REQUIRED - Create test suites. Minimum: basic unit tests for critical functions.

‚ú® Enhanced Workflows (Optional but Recommended)

These workflows add value and quality but are not mandatory for basic projects:

  • Prior Art Research (@Innovator): OPTIONAL - Research existing solutions and patents. Useful for avoiding reinvention.
  • Life Cycle Assessment (@Innovator): OPTIONAL - Environmental impact analysis. Recommended for production systems.
  • Requirements Version Control (@SeniorEng): OPTIONAL - Track requirement changes over time. Essential for complex/long projects.
  • Design Visualization & Feedback (@DesignEng): OPTIONAL - Multiple view angles, exploded views. Highly recommended for complex assemblies.
  • Part Numbering System (@DesignEng): OPTIONAL - Hierarchical part numbers. Required only if you need BOM management.
  • Design Revision Control (@DesignEng): OPTIONAL - Version history for designs. Recommended for iterative development.
  • Design Review Checkpoints (@DesignEng): OPTIONAL - Formal design reviews. Recommended for critical components.
  • Tolerance Stack-Up Analysis (@DesignEng): OPTIONAL - Critical dimension analysis. Required only for precision systems.
  • Design Compliance Matrix (@DesignEng): OPTIONAL - Verify design against requirements. Recommended for traceability.
  • Code Review Process (@Builder): OPTIONAL - Formal code reviews. Recommended for production code.
  • Test Coverage Metrics (@Skeptic): OPTIONAL - Coverage percentage tracking. Recommended for safety-critical systems.
  • Performance Benchmarking (@Skeptic): OPTIONAL - Performance baselines and trends. Recommended for real-time systems.

üè≠ Manufacturing Readiness (Conditional)

These workflows are required only if you plan to manufacture physical parts:

  • Manufacturing Documentation (@SeniorEng + @DesignEng): CONDITIONAL - Required for physical manufacturing. Includes technical drawings, GD&T, manufacturing notes.
  • Technical Drawings with GD&T (@DesignEng): CONDITIONAL - Required for manufacturing. Must include dimensions, tolerances, surface finish, threads.
  • Assembly Instructions (@DesignEng): CONDITIONAL - Required if assembling multiple parts.
  • Inspection Procedures (@Skeptic): CONDITIONAL - Required for quality control in manufacturing.
  • PMI Embedding (@Builder): CONDITIONAL - Optional enhancement for 3D annotated models.

üìä Project Dashboard

What is the Project Dashboard?

The Project Dashboard is an interactive HTML tool that provides a real-time visual overview of your project's status. It shows the current phase, requirements progress, design status, production readiness, change management, and compliance status - all from @SeniorEng's perspective.

üéØ Key Features

  • Real-Time Status: Automatically scans project files to determine current phase and progress
  • Requirements Tracking: Shows total requirements, approved, in-progress, and pending counts
  • Design Status: Tracks skeletons, manufacturing-ready files, drawings, DFM/DFA reviews
  • Production Readiness: Displays release gate status, version numbers, and documentation status
  • Change Management: Shows open ECOs, pending approvals, and approved changes
  • Compliance Status: Tracks compliance matrix, testing, and certification progress
  • Recent Activity: Displays latest project activities from session logs
  • Auto-Refresh: Updates every 60 seconds automatically
  • Manual Refresh: Click the refresh button anytime to update

üöÄ How to Use the Dashboard

1 Generate Dashboard Data

First, generate the dashboard data file by running the Python script:

python generate_dashboard_data.py

This creates dashboard_data.json which contains all the project status information.

2 Start a Local Web Server

‚ö†Ô∏è Important: The dashboard cannot load data when opened directly from the file system. You must use a web server.

Easy Method (Windows):

Double-click: start_dashboard_server.bat

PowerShell Method:

python -m http.server 8000

VS Code Method:

  • Install "Live Server" extension
  • Right-click PROJECT_DASHBOARD.html
  • Select "Open with Live Server"

‚ö†Ô∏è Keep the terminal window open while viewing the dashboard!

3 Open the Dashboard

Open your web browser and navigate to:

http://localhost:8000/PROJECT_DASHBOARD.html

The dashboard will automatically load and display your project status!

4 Refresh the Dashboard

When you make changes to your project:

  • Run python generate_dashboard_data.py to update the data
  • Click the "üîÑ Refresh Dashboard" button in the browser
  • Or wait for auto-refresh (happens every 60 seconds)

üîß How It Works

Data Generation

The generate_dashboard_data.py script:

  • Scans project files to determine current phase
  • Counts requirements and checks their status
  • Checks design file status (skeletons, manufacturing-ready, drawings)
  • Scans validation tools and reports (STEP validation, assembly validation)
  • Scans manufacturing feedback loop status (tools, data)
  • Scans production readiness files
  • Tracks change management (ECOs)
  • Checks compliance status
  • Reads recent activity from session logs
  • Generates dashboard_data.json with all this information

Dashboard Display

The PROJECT_DASHBOARD.html file:

  • Reads dashboard_data.json via HTTP (requires web server)
  • Displays data in interactive cards with color-coded status badges
  • Auto-refreshes every 60 seconds
  • Provides manual refresh button for immediate updates
  • Shows helpful error messages if data cannot be loaded
